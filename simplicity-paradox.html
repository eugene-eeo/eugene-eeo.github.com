The Simplicity Paradox

← Back to Index

Disclaimer:

I am not trying to pinpoint at any framework or project, but rather our "nature of coding" in general.

This article isn't trying to plead you to stop writing code or libraries- writing them and improving is part of programming, but rather as another random article on the internet.

So I began developing and using different libraries in Python and Perl (thought I do not publish the latter), and I've began to realize something: the libraries that declare themselves to be simple and minimal are eventually bloated.

Say I built a web framework- I'll name it Workman. I develop it over the course of a few months, adding features and cleaning up the code. I make a nice site, publish it, and get a large userbase.

Okay, so everybody's happy with the code, especially how minimal it is and how simple everything is- with little features and the literal meaning of micro hard-coded into it's design. Over the following months, I publish more releases, fixing bugs and adding features every time.

Given that a library is never "good enough", the future releases of Workman will keep adding more and more features, and there will come, at some point in time, where I cannot call it a "micro" framework anymore.

So, does this mean that developed libraries can never truly be "simple", as they claim to be? Every now and then a new library pops up, replaces another (usually more complex) one. It claims to be simple- and the developers add more and more features until one day the library becomes too heavy for practical use.

The cycle of development continues- a simple framework becomes bloated and cluttered with unnecessary features, and another one pops up to replace it. The said successor framework will then become bloated once more, like it's predecessor, and then will be replaced by another one. And the cycle is an infinite loop.

Should we stop writing code?

Yes and no. No, because competition and improvement is a major part of programming. Yes, because sometimes when you are developing a new library, do not try and replace whatever you can- doing so clutters your code. Rather, focus on your library's speciality and develop from there.

We do not need 50 web frameworks which all offer caching services- memcached and redis are tools which provide that service. We do not need frameworks which all replace another framework or tool at doing something. We do not need libraries which aim to do everything, because in the end they will do nothing.

Exceptions

While most Ruby frameworks try to wrap around Rack, their backbone webserver interface (with varying results), some do not. Most Python web frameworks also try to replace everything they find- some do it well, some don't.

Some good examples like Scorched and Flask are ideal frameworks, which do not aim to replace their underlying library- but serve as a high-level layer in which others can quickly build upon.

Another thing is to avoid using too much "magic", or magical functions which do things in the background. These things tend to clog up the developer's API without him knowing anything about it. As the Scorched documentation puts it: "No magicians welcomed".

Conclusion

Stop trying to replace everything you can since that's feature bloat and if you're developing on Python, you're violating the Zen of Python PEP. Stop trying to do everything since there is another tool for it- that's part of the Unix philosophy to use simple tools to solve problems.

Now that you've finished reading this article you can go ahead and try to replace another library or start your own trend. Best of luck for any ideas that you want to push into reality.
