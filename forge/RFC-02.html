considerations for plugins

As Forge will continue to grow, there would, at a certain point in time, come a need for plugins, or job-specific executables which can be preconfigured prior to running, see gruntjs.com/plugins for such examples.

There are several ways in which these can be implemented- firstly, they can be stored in some git repo and then downloaded when needed, ran, and then deleted. This would bring about multiple issues like unstable builds, bandwidth issues, etc.

Secondly, plugins can be stored in either the user's home folder, which would now depend on the os.getenv("HOME") output, or a "lower-level" folder like /usr/local/.

invocation methods

There are also multiple ways in which the plugins can be invoked- the command-line method or the config-file method. Both have their advantages and disadvantages.

The command-line method involves the invocation of the plugin's main executable(s) using command-line options which would, presumably, be parsed by the executable itself.

An alternative method might be to create a static configuration file inside a folder like /usr/local/forge/config/, which would presumably be parsed by the plugin's executable, and then invoke the executable.

With my current coding skills and forge's relatively new age, it seems best that we would go for a static configuration-file based invocation, and a directory for storing plugins- /usr/local/ would seem the rational way to go.
