express.contrib.HTMLEscape(string)

Used in the tinywerk app, this function will take a string and return a HTML-escaped version of it, which will replace all “<” with “&lt;”, “>” with “&gt;”, and “&” with “&amp;” to prevent forging of data.

express.contrib.parseMonth(month)

Used in the tinywerk app. The function takes in a month which is gained by splitting the value from the string returned by time.asctime(). Returns the month part of the formula implemented in the ProcessTime function.

express.contrib.ProcessTime(t)

Takes in a string which is a return of the time.asctime() method and returns the time in the following formula:

>>> t = time.asctime()
>>> print(t)
Mon Aug 19 20:38:33 2013
>>> express.contrib.ProcessTime(t)
20130819203833

This is useful when indexing posts or any arbitrary data that inserting needs to be keep tracked with a asctime string. Was used because it was too late to change my database schema.

class express.contrib.Cache

Cheap alternative to werkzeug.contrib.cache.SimpleCache by having fairly simple internals and custom timeouts.

Cache.expire(key)

Explicityly expires a key from the cache by removing it. If the key doesn’t exist, it will silently fail while returning True even if it failed.

Cache.get(key, default=None)

Get the item named key within the cache, given that it is found and it hasn’t expired. If either of the above two conditions are not present, it will return the defaut value which defaults to None.

Cache.keys()

Gets the list of keys within the cache and returns it as a list and not an iterable.

Cache.set(key, data, timeout=300)

Set a key named after key within the cache. The default timeout is 5 minutes but you may want to change it according to the key’s importance. You may also want to set timeout to “forever” to preserve the data for as long as the application runs.

class express.contrib.Bucket

Load-sharing storage made for the expressDB storage API which made it possible to use multiple redis instances and balance the data between them; i.e. if one of the instances failed we will still have most of our data and no significant loss.

Basically is a high-level sharding layer for the underlying redis databases that is resistant to full databases and or any errors thrown during IO operations.

Bucket.get(key)

Get a key in the first database which it appears/ was found. The database’s get method must be resistant to KeyErrors and must also support returning None values if the key isn’t found, which is something found in most database engines.

Bucket.gethash(key)

Similar to the get hash method of the express.helpers.RedisStorage object, but will return a unique key which is unique across all of the databases.

Bucket.keys()

Get the keys within the current bucket, by compiling the keys of all of the linked databases.

Bucket.rem(key)

Removes a key in the first database in which it was found.

Bucket.set(key, data)

Sets a key in the next database which is determined by the next_db property of the Bucket object. This determines which database to write to and is one of the most basic forms of load sharding.

Returns True if the data was written, False if it wasn’t.

Bucket.setup(storage, from_list=[])

Sets up the Bucket() object. storage is an arbitrary storage class that takes in the arguments specified by the list of dictionaries in from_list, i.e.:

import express my_db_list = [{“host”:”localhost”, “port”:8911}]

store = Bucket() store.setup(express.helpers.RedisStorage, my_db_list)

class express.contrib.Engine

Conveinience function not meant as a replacement for jinja2 but rather as a smaller cannon to be used if the big guns are albeit too overkill.

This class was used in expressDB as part of it’s page rendering API. Most of the code was copied except for the code generating the navigation bars.

Not many functions are defined here so you may want to subclass this method to suit your needs.

Engine.render(**kwargs)

Render the template either with our extremely dumb fallback method or the pystache’s default rendering method. The extremely dumb method processes the template like so:

def render(template, arguments):
    return template.format(**arguments)

But the smarter pystache-mustache implementation uses default mustache syntax to generate the template.

It accepts any number of keyword arguments which will be made into a dictionary and passed to the rendering function as the second argument.

Engine.setup(template)

Sets up the Engine() object, accepts a file named template which will be used as the base template. This is the concept which both tinywerk and expressDB is based on.
